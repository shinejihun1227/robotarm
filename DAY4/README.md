## Lesson 1. 카메라 스트리밍 기초

### 도입
로봇팔에 "눈"을 달아주는 첫 단계다. 사람이 손으로 물체를 집을 때 먼저 눈으로 위치를 파악하듯, 로봇팔도 카메라로 세상을 봐야 자동으로 동작할 수 있다. 이번 레슨에서는 카메라 영상을 PC로 가져오는 가장 기본적인 방법을 익힌다.

### 핵심 개념

**영상은 사진의 연속이다**
동영상은 1초에 수십 장의 사진(프레임)을 빠르게 보여주는 것이다. 카메라 프로그래밍의 본질은 "프레임을 한 장 받아서 → 처리하고 → 화면에 보여주는" 루프를 빠르게 반복하는 것이다.

```
[프레임 받기] → [처리] → [화면 표시] → [다시 처음으로]  ← 1초에 30번 반복
```

**IP 카메라 vs USB 웹캠**
- USB 웹캠: `cv2.VideoCapture(0)` — 숫자로 연결
- 휴대폰 IP 카메라: `cv2.VideoCapture('http://폰IP:포트/video')` — 주소로 연결
- 휴대폰을 카메라로 쓰면 별도 장비 없이 위에서 내려다보는 구도를 만들기 쉽다.

### 코드 분석 — `camera_test.py`

```python
cap = cv2.VideoCapture(url)        # ① 스트림 연결
while True:
    ret, frame = cap.read()        # ② 프레임 한 장 받기
    if not ret:                    # ③ 실패하면 중단
        break
    cv2.imshow('Camera Test', frame)   # ④ 화면에 표시
    if cv2.waitKey(1) & 0xFF == ord('q'):  # ⑤ q 누르면 종료
        break
cap.release()                      # ⑥ 자원 정리
cv2.destroyAllWindows()
```

- `cap.read()` 는 `(성공여부, 프레임)` 두 개를 동시에 돌려준다. 그래서 `ret, frame` 으로 받는다.
- `waitKey(1)` 은 1ms 동안 키 입력을 기다린다. 이게 없으면 화면이 안 뜬다 (영상 루프의 필수 요소).
- `release()` 와 `destroyAllWindows()` 를 안 하면 카메라가 다음 실행 때 안 잡힐 수 있다.

### 실습 단계
1. 휴대폰에 IP 카메라 앱(IP Webcam, DroidCam 등) 설치
2. 휴대폰과 PC를 **같은 WiFi**에 연결
3. 앱이 표시하는 주소를 `camera_test.py` 의 `url` 에 입력
4. 실행 → 영상이 뜨면 성공, `q` 로 종료

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| "프레임을 읽을 수 없습니다" | 주소 오류 / 다른 네트워크 | 휴대폰·PC 같은 WiFi 확인, 주소 재확인 |
| 영상이 느리고 끊김 | 해상도 높음 / 신호 약함 | 앱에서 해상도 낮추기 |
| 창이 안 뜸 | `waitKey` 누락 | 영상 루프에 반드시 포함 |

### 정리
"프레임 받기 → 처리 → 표시" 루프가 모든 비전 코드의 뼈대다. 다음 레슨부터 ②와 ④ 사이의 "처리" 부분을 채워나간다.

## Lesson 2. 색상 기반 물체 인식 (HSV)

### 도입
카메라가 영상을 보내주지만, 컴퓨터에게 "파란 물체가 어디 있는지"는 알려주지 않는다. 이번 레슨에서는 영상에서 특정 색을 가진 물체를 찾아 그 중심 좌표를 계산한다.

### 핵심 개념

**왜 RGB가 아니라 HSV인가**
RGB는 빨강·초록·파랑의 양으로 색을 표현한다. 문제는 조명이 바뀌면 세 값이 전부 흔들린다는 것이다. HSV는 색을 세 가지로 분리한다.

- **H (Hue, 색상)**: 빨강·파랑 같은 "색 자체"
- **S (Saturation, 채도)**: 색의 진하기
- **V (Value, 명도)**: 밝기

조명이 바뀌면 주로 **V만 변하고 H는 거의 그대로**다. 그래서 "파란색"을 H 범위로 지정하면 조명이 달라져도 안정적으로 잡힌다.

> **중요**: OpenCV에서 H는 0~179 범위다 (0~255가 아님!). S와 V는 0~255다. 이걸 헷갈리면 색 범위 지정이 어긋난다.

**물체 중심을 찾는 4단계**
```
① inRange   : 지정한 색 범위만 흰색으로 (마스크 생성)
② 모폴로지   : 작은 노이즈 제거, 구멍 메우기
③ 컨투어     : 흰 영역들의 외곽선 추출
④ 무게중심   : 가장 큰 영역의 중심 좌표 계산
```

- **모폴로지 OPEN**: 작은 흰 점(노이즈) 제거
- **모폴로지 CLOSE**: 물체 안의 작은 검은 구멍 메우기
- **무게중심**: `cx = M10/M00`, `cy = M01/M00` (moments 공식)

### 코드 분석 — `color_detector.py`

```python
hsv  = cv2.cvtColor(frame, cv2.COLOR_BGR2HSV)      # BGR → HSV 변환
mask = cv2.inRange(hsv, COLOR_LOWER, COLOR_UPPER)  # 색 범위만 추출

kernel = np.ones((5, 5), np.uint8)
mask   = cv2.morphologyEx(mask, cv2.MORPH_OPEN,  kernel)  # 노이즈 제거
mask   = cv2.morphologyEx(mask, cv2.MORPH_CLOSE, kernel)  # 구멍 메우기

contours, _ = cv2.findContours(mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
largest = max(contours, key=cv2.contourArea)        # 가장 큰 영역
if cv2.contourArea(largest) < 500:                  # 너무 작으면 무시
    return None
M  = cv2.moments(largest)
cx = int(M['m10'] / M['m00'])                       # 중심 x
cy = int(M['m01'] / M['m00'])                       # 중심 y
```

- `contourArea < 500` 조건: 작은 노이즈를 물체로 오인하지 않게 면적 하한선을 둔다.
- 가장 큰 컨투어만 선택하므로, 화면에 같은 색이 여러 개면 가장 큰 것만 잡힌다.

### 실습 단계
1. `color_hsv_test.py` 실행 → 화면의 물체를 **마우스로 클릭**
2. 콘솔에 출력되는 `COLOR_LOWER` / `COLOR_UPPER` 값 확인
3. 그 값을 `config_vision.py` 에 입력
4. `blue_hsv_test.py` 실행 → 오른쪽 마스크 화면을 보며 범위 미세 조정
   - 물체만 하얗게, 배경은 까맣게 나오도록 조정

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 물체가 안 잡힘 | H 범위 너무 좁음 | 클릭으로 H값 재확인 후 범위 넓히기 |
| 배경까지 잡힘 | S/V 하한 너무 낮음 | 채도·명도 하한 올리기 |
| 물체가 끊겨 보임 | 노이즈 | 커널 크기(5,5)→(7,7) 키우기 |

### 정리
이제 영상에서 물체의 **픽셀 좌표 (cx, cy)** 를 얻을 수 있다. 하지만 픽셀은 로봇이 이해 못 한다. 다음 단계는 "실제 거리(cm)"로 바꾸는 것이다.

## Lesson 3. ArUco 마커 인식

### 도입
물체의 픽셀 위치를 알았지만, 카메라가 멀리 있으면 작게, 가까이 있으면 크게 보인다. "화면의 1픽셀이 실제로 몇 cm인지" 기준이 필요하다. 그 기준을 만들어주는 것이 ArUco 마커다.

### 핵심 개념

**ArUco 마커란**
흑백 격자 패턴으로 된 정사각형 마커다. QR코드처럼 ID를 담고 있고, 카메라로 위치·크기·방향을 정확히 인식할 수 있다.

- `DICT_4X4_50`: 내부 4×4 격자, 총 50종류의 마커를 구분
- 각 마커는 고유 ID(0, 1, 2…)를 가진다.

**왜 마커로 스케일을 구하나**
마커의 **실제 한 변 길이(cm)** 를 미리 알고 있으면, 화면에서 그 변이 몇 픽셀인지 재서 비율을 계산할 수 있다.

```
px_per_cm = (마커 한 변의 픽셀 길이) / (실제 cm)

예: 마커가 화면에서 80px, 실제 4cm
    → 80 / 4 = 20 px/cm
    → 화면의 20픽셀 = 실제 1cm
```

마커 0번을 **좌표 원점**으로 삼으면, 모든 물체 위치를 마커 기준 상대 위치로 표현할 수 있다.

### 코드 분석 — `aruco_detector.py`

```python
# 마커 감지
gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)     # 흑백 변환 (마커는 흑백)
corners, ids, _ = self.detector.detectMarkers(gray)

# 각 마커의 중심 계산
for i, mid in enumerate(ids.flatten()):
    c  = corners[i][0]               # 마커 네 꼭짓점
    cx = int(c[:, 0].mean())         # 꼭짓점 x들의 평균 = 중심 x
    cy = int(c[:, 1].mean())         # 중심 y

# 스케일 계산 (get_scale)
side_px   = np.linalg.norm(corners[0] - corners[1])  # 한 변의 픽셀 길이
px_per_cm = side_px / cfg.MARKER_CM                  # 픽셀당 cm
origin_px = (markers[0][0], markers[0][1])           # 마커0 = 원점
```

- 마커는 흑백 패턴이라 `COLOR_BGR2GRAY` 로 흑백 변환 후 인식한다.
- `corners[0] - corners[1]` 은 마커 첫 변의 벡터, `np.linalg.norm` 으로 그 길이(픽셀)를 구한다.

### 실습 단계
1. ArUco 마커(`DICT_4X4_50`, ID 0번)를 인쇄
2. **자로 실제 한 변 길이를 측정** → `config_vision.py` 의 `MARKER_CM` 에 입력 (매우 중요)
3. `aruco_test.py` 실행 → 마커를 비추면 ID와 스케일이 화면에 표시됨
4. 마커를 멀리·가까이 움직여보며 스케일이 어떻게 변하는지 관찰

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 마커 인식 안 됨 | 인쇄 흐림 / 조명 반사 | 선명하게 인쇄, 무광 용지, 반사 줄이기 |
| 스케일이 흔들림 | 마커 기울어짐 | 마커를 카메라와 수평으로 평평하게 |
| 거리 계산 틀림 | `MARKER_CM` 오입력 | 실제 인쇄된 크기 재측정 |

### 정리
이제 px_per_cm(스케일)과 원점(마커0 위치)을 얻었다. Lesson 2의 픽셀 좌표 + 이번 스케일 = 실제 거리 변환이 가능해진다. 다음 레슨에서 합친다.

## Lesson 4. 좌표계 변환 (픽셀 → 로봇)

### 도입
드디어 두 정보를 합친다. 물체의 픽셀 좌표(L2)와 스케일·원점(L3)을 이용해, 로봇팔이 이해하는 좌표(cm)로 변환한다. 이 레슨의 핵심은 "카메라가 보는 세상"과 "로봇이 사는 세상"의 좌표계가 다르다는 점이다.

### 핵심 개념

**세 개의 좌표계**
```
① 픽셀 좌표   : 화면 왼쪽 위가 (0,0), 오른쪽·아래로 갈수록 증가
② 마커 좌표   : 마커0을 원점으로 한 상대 위치 (cm)
③ 로봇 좌표   : 로봇팔 베이스가 원점, 앞/옆/위 방향
```

**변환 2단계**
```
1단계: 픽셀 → 마커 기준 cm
   dx_cm = (물체px - 원점px) / px_per_cm

2단계: 마커 cm → 로봇 cm
   카메라가 위에서 내려다보므로 축이 회전·반전됨
   + 마커와 로봇 베이스 거리(BASE_OFFSET) 보정
```

**왜 dx, dy가 뒤바뀌나 (핵심 토론)**
카메라를 로봇팔 위쪽에서 아래로 설치하면, 화면의 좌우(가로)가 로봇의 좌우가 되고, 화면의 상하(세로)가 로봇의 앞뒤가 된다. 그래서 코드에서 화면 세로 거리(dy)가 로봇 앞뒤(robot_x)에 들어간다.

### 코드 분석 — `vision_test.py` 의 `pixel_to_robot`

```python
def pixel_to_robot(px, py, px_per_cm, origin_px):
    dx_cm = (px - origin_px[0]) / px_per_cm    # 마커 기준 가로 거리
    dy_cm = (py - origin_px[1]) / px_per_cm    # 마커 기준 세로 거리

    robot_x = cfg.BASE_OFFSET_X - dy_cm        # 세로 → 로봇 앞뒤
    robot_y = (dx_cm + cfg.BASE_OFFSET_Y)      # 가로 → 로봇 좌우
    return round(robot_x, 1), round(robot_y, 1)
```

- `BASE_OFFSET_X/Y` 는 마커 원점과 로봇 베이스 사이의 거리다 (다음 레슨에서 측정).
- 부호(`-dy_cm` 등)는 카메라 설치 방향에 따라 달라진다. 팔이 반대로 움직이면 부호를 뒤집는다.

> **주의**: 실습 코드들(`vision_test.py`, `aruco_detector.py`, `main_vision2.py`)마다 부호 처리가 조금씩 다르다. 이는 카메라 설치 방향을 바꿔가며 맞춘 흔적이다. 학습자에게 "정답 부호는 없고, 본인 세팅에 맞춰 실험으로 찾는다"는 점을 강조한다.

### 실습 단계
1. `vision_test.py` 실행 → 마커 + 파란 물체를 동시에 비추기
2. 물체를 오른쪽으로 옮길 때 robot_y가 직관과 같은 방향으로 변하는지 확인
3. 반대로 움직이면 `pixel_to_robot` 의 해당 부호를 뒤집기
4. 앞뒤(robot_x)도 같은 방식으로 검증

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 팔이 좌우 반대로 감 | robot_y 부호 | `dx_cm` 부호 뒤집기 |
| 팔이 앞뒤 반대로 감 | robot_x 부호 | `dy_cm` 부호 뒤집기 |
| 좌표가 전체적으로 어긋남 | BASE_OFFSET 미보정 | Lesson 6 캘리브레이션 필요 |

### 정리
이제 "물체가 로봇 기준 어디에 있는지"를 cm로 계산할 수 있다. 하지만 정확하려면 마커-로봇 거리(BASE_OFFSET)를 정밀히 맞춰야 한다. 그전에, 계산한 좌표를 ESP32에 보내는 방법(통신)을 먼저 익힌다.

## Lesson 5. PC ↔ ESP32 시리얼 통신

### 도입
PC가 물체 좌표를 계산했지만, 실제로 움직이는 것은 ESP32에 연결된 로봇팔이다. PC의 계산 결과를 ESP32로 전달해야 한다. 가장 기본적이고 확실한 방법인 **USB 시리얼 통신**을 배운다. 이 레슨은 PART 2(통신) 전체의 출발점이기도 하다.

### 핵심 개념

**시리얼 통신이란**
USB 케이블을 통해 데이터를 **한 줄(line)씩** 주고받는 방식이다. Thonny가 ESP32와 대화하는 것도 바로 이 방식이다.

```
PC ──USB──→ ESP32   : 명령 전송  {"cmd":"move_to","x":15,...}
PC ←──USB── ESP32   : 응답 수신  {"ok":true}
```

**baudrate (통신 속도)**
양쪽이 똑같은 속도로 맞춰야 대화가 된다. 보통 `115200`을 쓴다. PC와 ESP32 코드 양쪽이 같은 값이어야 한다.

**JSON 명령 프로토콜**
명령을 사람·기계 모두 읽기 쉬운 JSON으로 약속한다.
```
이동:   {"cmd": "move_to", "x": 15, "y": 0, "z": 20}
홈:     {"cmd": "home"}
그리퍼: {"cmd": "grip", "angle": 50}
```

### 코드 분석 — `serial_comm.py`

```python
class ArmSerial:
    def __init__(self, port, baudrate=115200):
        self.ser = serial.Serial(port, baudrate, timeout=3)
        time.sleep(2)              # ESP32 부팅 대기 (중요)

    def send(self, cmd: dict) -> dict:
        self.ser.reset_input_buffer()       # 이전 찌꺼기 데이터 비우기
        msg = json.dumps(cmd) + '\n'        # 딕셔너리 → JSON 문자열 + 줄바꿈
        self.ser.write(msg.encode())        # 전송
        resp = self.ser.readline().decode().strip()  # 응답 한 줄 읽기
        return json.loads(resp)

    def move_to(self, x, y, z, grip=None):
        cmd = {'cmd': 'move_to', 'x': x, 'y': y, 'z': z}
        return self.send(cmd)
```

- `time.sleep(2)`: ESP32는 연결 직후 부팅 중이라 바로 명령을 받지 못한다. 잠깐 기다린다.
- `reset_input_buffer()`: 이전에 쌓인 응답 찌꺼기를 비워서, 이번 명령의 응답만 깨끗하게 받는다.
- `'\n'` (줄바꿈): ESP32가 "명령이 여기서 끝났다"를 알 수 있게 하는 구분자.
- 명령마다 메서드(`move_to`, `home`, `grip_open`)로 추상화 → 사용자는 JSON을 몰라도 됨.

### 실습 단계
1. ESP32에 JSON 명령을 받아 `ik.ik()` 를 실행하는 펌웨어 준비
2. 장치 관리자(Windows) 또는 `ls /dev/tty*`(Mac/Linux)로 포트 확인
3. `config_vision.py` 의 `SERIAL_PORT` 설정 (예: `COM7`)
4. Python에서 직접 테스트:
   ```python
   from serial_comm import ArmSerial
   arm = ArmSerial('COM7')
   arm.home()
   arm.move_to(15, 0, 20)
   ```

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 포트 연결 실패 | Thonny가 포트 점유 중 | **Thonny를 닫고** 실행 (포트는 하나만 점유 가능) |
| 응답 없음 | baudrate 불일치 | 양쪽 115200 확인 |
| 명령 씹힘 | 부팅 대기 부족 | `sleep(2)` 유지 |

### 정리 — 통신 트랙으로 가는 다리
여기서 배운 "JSON 명령을 보내고 응답을 받는다"는 개념은 PART 2에서 **그대로** 쓰인다. WiFi든 WebSocket이든 MQTT든, 바뀌는 것은 "USB선" → "무선"이라는 전달 수단뿐이고, 명령 구조는 동일하다.

## Lesson 6. 캘리브레이션 (좌표 보정)

### 도입
Lesson 4의 변환에는 `BASE_OFFSET_X/Y`(마커-로봇 거리)가 들어갔다. 이 값이 부정확하면 로봇팔이 물체 옆을 헛집는다. 이번 레슨에서는 이 값을 실측으로 정밀하게 맞추는 캘리브레이션을 수행한다.

### 핵심 개념

**캘리브레이션 = 두 좌표계 정렬**
"카메라가 본 좌표"와 "로봇이 실제 도달하는 좌표"를 일치시키는 작업이다. 방법은 역으로 추적하는 것이다.

```
1. 로봇팔을 홈으로 보낸다 (로봇 좌표를 아는 상태)
2. 그리퍼 끝 바로 아래에 물체를 놓는다
3. 카메라로 그 물체의 마커 기준 위치를 측정한다
4. "로봇이 아는 위치" - "카메라가 본 위치" = OFFSET
```

**여러 번 측정 → 평균**
카메라 측정엔 노이즈가 있으므로, 3~5회 측정 후 평균을 내어 안정적인 값을 얻는다.

**설정 자동 반영**
측정 결과를 사람이 손으로 옮겨 적으면 실수하기 쉽다. 정규식으로 `config_vision.py` 의 값을 코드가 직접 수정한다.

### 코드 분석 — `calibration.py`

```python
# SPACE 누를 때마다 측정
dx_cm = (obj_px[0] - origin_px[0]) / px_per_cm
dy_cm = (obj_px[1] - origin_px[1]) / px_per_cm

robot_home_x = 19.5    # fk(0,0,0)의 x — 홈에서 그리퍼의 실제 위치
offset_x = robot_home_x + dy_cm
offset_y = 0.0 - dx_cm
offset_samples.append((offset_x, offset_y))   # 측정값 누적

# S 누르면 평균내서 저장 (save_config)
content = re.sub(r'BASE_OFFSET_X\s*=\s*[-\d.]+',
                 f'BASE_OFFSET_X = {round(offset_x, 1)}', content)
```

- `re.sub`: `config_vision.py` 파일에서 `BASE_OFFSET_X = 숫자` 패턴을 찾아 새 값으로 교체.
- `calibration.json` 에도 백업 저장 → 나중에 복구·기록 용도.

### 실습 단계
1. 로봇팔 홈으로 이동
2. 그리퍼 끝 정확히 아래에 파란 물체 배치
3. `calibration.py` 실행 → `SPACE` 를 3~5회 눌러 측정
4. `S` 키로 저장 (config 자동 수정됨)
5. `vision_test.py` 재실행 → 좌표 정확도 개선 확인

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 측정값이 매번 크게 다름 | 마커/물체 흔들림 | 고정 후 측정, 측정 횟수 늘리기 |
| `robot_home_x` 안 맞음 | FK 홈값 다름 | 본인 `ik.fk(0,0,0)` 결과로 교체 |
| 저장 안 됨 | 파일 권한/경로 | 실행 폴더에 config 있는지 확인 |

### 정리
이제 카메라 좌표와 로봇 좌표가 정렬되어, 본 곳을 정확히 집을 수 있다. 모든 부품(인식·변환·통신·보정)이 준비됐다. 다음 레슨에서 하나로 합친다.

## Lesson 7. 통합 — 자동 Pick & Place

### 도입
지금까지 만든 모듈을 전부 결합해 "카메라로 본 물체를 자동으로 집는" 완성 시스템을 만든다. 동시에, 로봇이 움직이는 동안 화면이 멈추지 않게 하는 멀티스레딩이라는 실전 기법을 배운다.

### 핵심 개념

**전체 파이프라인**
```
영상 → ArUco 감지 → 색상 감지 → 좌표 변환 → 시리얼 전송 → ESP32 → 동작
```

**왜 단계적으로 하강하나**
물체로 한 번에 내려가면 충돌·오차 위험이 크다. 그래서 안전하게 나눈다.
```
HOVER_Z(높이 22cm) → 중간 높이 → PICK_Z(집는 높이 10cm)
   물체 위로 접근       천천히        실제 집기
```

**왜 멀티스레딩이 필요한가**
로봇팔 이동은 몇 초 걸린다. 이동을 메인 루프에서 직접 실행하면 그동안 영상이 얼어붙는다. 그래서 이동 동작을 **별도 스레드**에서 돌려, 영상은 계속 흐르게 한다.

### 코드 분석 — `main_vision2.py`

```python
is_running = False     # 동작 중 플래그

def pick_and_place(robot_x, robot_y):
    global is_running
    is_running = True              # 동작 시작 표시
    arm.move_to(robot_x, robot_y, cfg.HOVER_Z)   # ① 물체 위로
    time.sleep(3.0)
    arm.move_to(robot_x, robot_y, mid_z)         # ② 중간 높이
    arm.move_to(robot_x, robot_y, cfg.PICK_Z)    # ③ 집는 높이
    arm.home()                                   # ④ 홈 복귀
    is_running = False             # 동작 완료

# 메인 루프에서 SPACE 누르면:
if key == ord(' '):
    if is_running:
        print('동작 중 - 기다려주세요')     # 중복 방지
    elif robot_xy is None:
        print('물체 또는 마커 없음')
    else:
        t = threading.Thread(target=pick_and_place, args=(robot_xy[0], robot_xy[1]))
        t.daemon = True
        t.start()                  # 별도 스레드로 동작 실행
```

- `is_running` 플래그: 동작 중에 SPACE를 또 눌러도 명령이 겹치지 않게 막는다.
- `threading.Thread`: 집기 동작을 백그라운드로 보내 영상 루프를 살린다.
- `daemon = True`: 메인 프로그램 종료 시 스레드도 함께 종료.

### 실습 단계
1. 마커 + 물체를 비추고 화면에 로봇 좌표가 뜨는지 확인
2. `SPACE` → 로봇팔이 물체 위로 접근 → 단계적 하강 관찰
3. `H`(홈), `Q`(홈 복귀 후 종료) 동작 확인
4. **확장 과제**: 집은 뒤 `PLACE_POSITION` 으로 옮겨 놓는 동작 추가

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 동작 중 영상 멈춤 | 스레드 미사용 | `threading` 으로 분리 확인 |
| 물체 옆을 헛집음 | 캘리브레이션 부정확 | Lesson 6 재수행 |
| 명령 겹쳐 오작동 | 플래그 미확인 | `is_running` 체크 유지 |

### 최종 과제
카메라로 본 임의 위치의 물체를 집어 지정된 곳에 놓는 **완전 자동 Pick & Place** 완성. 인식·변환·통신·보정·동작이 모두 맞물려야 성공한다.

## Lesson 8. 통신 개요 — 무엇을 언제 쓸까

### 도입
Lesson 5에서 USB 시리얼로 ESP32에 명령을 보냈다. 이제 그 "전달 수단"을 다양하게 확장한다. 그 전에, 통신을 큰 그림으로 분류해 이해한다.

### 핵심 개념

**통신을 계층으로 보기**
```
응용 계층   HTTP / WebSocket / MQTT     ← 무엇을 어떻게 주고받을지
─────────────────────────────────
전송 계층   TCP / UDP                   ← 어떻게 안전하게 보낼지
```

**TCP vs UDP**

| 구분 | TCP | UDP |
|---|---|---|
| 연결 | 연결 확립 후 통신 | 연결 없이 바로 전송 |
| 신뢰성 | 도착·순서 보장 | 보장 없음 |
| 속도 | 상대적으로 느림 | 빠름 |
| 비유 | 등기우편(수령확인) | 일반 전단지(뿌리기) |
| 용도 | 명령 제어, 웹, 파일 | 게임, 영상, 센서 스트림 |

**무선 수단 비교**

| 수단 | 거리 | 인터넷 | 특징 |
|---|---|---|---|
| 시리얼(UART) | 케이블 | ✕ | 가장 단순·확실 |
| WiFi | 수십 m | ○ | 빠름, 전력 많이 씀 |
| BLE | 10~30m | ✕ | 저전력, 앱 연결 |
| ESP-NOW | ~200m | ✕ | ESP32끼리 직접 |

### 토론
"로봇팔 명령 제어"에는 신뢰성(TCP)이 중요할까, 속도(UDP)가 중요할까? → 명령이 누락되면 위험하므로 보통 TCP 계열을 쓴다. 단, 실시간 연속 입력(조이스틱)에는 UDP가 유리할 수 있다.

### 정리
앞으로 배울 HTTP·WebSocket·MQTT는 모두 TCP 위에서 동작한다. 각각 "연결 방식"과 "용도"가 다를 뿐, Lesson 5의 JSON 명령 개념은 그대로 이어진다.

## Lesson 9. WiFi 기초 — AP / STA 모드와 HTTP 제어

### 도입
USB선을 끊고 무선으로 넘어간다. ESP32는 WiFi가 내장되어 있어 별도 부품 없이 무선 제어가 가능하다. 브라우저만으로 로봇팔을 조종해본다.

### 핵심 개념

**WiFi 3가지 모드**
```
AP 모드     : ESP32가 공유기 역할 (다른 기기가 ESP32에 접속)
              인터넷 X, 공유기 X, 현장 직접 제어에 최적

STA 모드    : ESP32가 기존 공유기에 연결 (일반 기기처럼)
              인터넷 O, 실제 IoT 제품 방식

AP+STA      : 두 가지 동시 (게이트웨이/설정 페이지용)
```

**HTTP 요청-응답 구조**
```
브라우저 ──요청(GET/POST)──→ ESP32 웹서버
         ←──응답(HTML/JSON)──
         (한 번 주고받고 연결 끊김)
```

**핵심 주의 — ADC2 핀 제약**
ESP32의 ADC2는 **WiFi와 회로를 공유**해서, WiFi를 켜면 ADC2를 못 쓴다.
```
ADC1 (WiFi와 함께 OK) : GPIO 32, 33, 34, 35, 36, 39
ADC2 (WiFi 켜면 불가)  : GPIO 0, 2, 4, 12, 13, 14, 15, 25, 26, 27
```
현재 `config.py` 의 가변저항 핀(`POT_PINS = [13,12,14,27,26,25]`)이 **전부 ADC2**다. 가변저항 제어와 WiFi를 동시에 쓰려면 ADC1 핀으로 옮겨야 한다.

### 실습 단계
1. ESP32를 AP 모드로 실행 → 휴대폰 WiFi에서 `RobotArm` 접속
2. 브라우저에서 `192.168.4.1` 접속 → 슬라이더로 XYZ·그리퍼 제어
3. **충돌 실험**: 가변저항 제어 코드와 WiFi를 동시에 켜서 ADC2가 먹통 되는지 확인
4. STA 모드로 바꿔 학교 WiFi에 연결, 같은 네트워크 PC에서 접속

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 가변저항 값 0 고정 | ADC2 + WiFi 충돌 | ADC1(32~39)로 핀 이동 |
| AP 접속 안 됨 | 비밀번호 8자 미만 | WPA2는 8자 이상 필요 |
| 부팅 이상 | GPIO12 풀업 | 스트랩핑 핀 회로 점검 |

### 정리
무선으로 로봇팔을 제어하는 데 성공했다. 하지만 HTTP는 버튼 누를 때마다 재연결한다. 실시간 제어엔 아쉽다. 다음 레슨에서 연결을 유지하는 방식을 배운다.

## Lesson 10. WebSocket — 실시간 양방향 제어

### 도입
HTTP로 슬라이더를 움직이면 매번 버튼을 눌러 요청해야 한다. 실시간 조종 느낌이 안 난다. WebSocket은 연결을 끊지 않고 유지해, 슬라이더를 움직이는 즉시 로봇이 따라오게 만든다.

### 핵심 개념

**HTTP vs WebSocket**
```
HTTP        : 요청 → 응답 → 끊김 (매번 새 연결)
WebSocket   : 한 번 연결 → 계속 유지 → 양방향 자유 통신
```

**핸드셰이크 (연결 전환)**
WebSocket은 HTTP로 시작해서 중간에 프로토콜을 바꾼다.
```
브라우저: "Upgrade: websocket" (HTTP 말고 WebSocket으로 바꾸자)
ESP32:    "101 Switching Protocols" (좋아, 바꾸자)
이후로 WebSocket 프레임으로 통신
```

**양방향의 의미**
HTTP는 브라우저가 물어봐야 ESP32가 답한다. WebSocket은 **ESP32가 먼저** 현재 좌표를 보낼 수 있다. 로봇 상태를 실시간 표시할 때 유리하다.

### 실습 단계
1. ESP32에 WebSocket 서버 구현 (MicroPython용 라이브러리 또는 직접 핸드셰이크)
2. 브라우저 슬라이더를 움직이는 즉시 로봇팔이 따라오게 연결
3. ESP32가 현재 좌표를 주기적으로 브라우저에 푸시 → 화면에 실시간 표시
4. **비교 실험**: 같은 동작을 HTTP / WebSocket으로 제어해 반응 지연 차이 측정

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 연결 곧 끊김 | ping/pong 미처리 | 주기적 ping으로 연결 유지 |
| 명령 폭주 | 슬라이더마다 전송 | 일정 간격(throttle)으로 제한 |

### 정리
이제 부드러운 실시간 무선 제어가 가능하다. 하지만 둘 다 "같은 네트워크 안"에서만 동작한다. 인터넷 너머 원격 제어는 다음 레슨에서 다룬다.

---

## Lesson 11. MQTT — 클라우드 연동·원격 제어

### 도입
지금까지는 ESP32와 제어 기기가 같은 공간·네트워크에 있어야 했다. MQTT를 쓰면 인터넷을 통해 집에서 학교 로봇팔을, 또는 여러 대를 한 번에 제어할 수 있다. 실제 IoT 제품에서 가장 널리 쓰는 방식이다.

### 핵심 개념

**발행/구독 모델 (Pub/Sub)**
직접 연결이 아니라 **브로커(중계 서버)** 를 거친다.
```
제어기 ──publish──→ [브로커] ──→ subscribe ──→ ESP32
                  (중계 서버)
```
- 발행자와 구독자는 서로의 주소를 몰라도 된다. 브로커만 알면 된다.
- 한 메시지를 여러 구독자가 동시에 받을 수 있다 (다중 기기 제어).

**토픽(Topic)**
메시지를 주제별로 분류하는 주소다.
```
robotarm/cmd      ← 명령을 보내는 토픽 (ESP32가 구독)
robotarm/status   ← 상태를 알리는 토픽 (ESP32가 발행)
```

**무료 브로커**
HiveMQ 등 공개 브로커를 쓰면 서버 구축 없이 바로 실습 가능하다.

### 실습 단계
1. ESP32를 STA 모드로 인터넷 연결 후 MQTT 브로커에 접속
2. ESP32가 `robotarm/cmd` 토픽 구독 → 명령 수신 시 동작
3. 휴대폰 MQTT 앱(IoT MQTT Panel 등)에서 명령 발행 → 로봇팔 동작
4. ESP32가 현재 좌표를 `robotarm/status` 로 주기적 발행
5. **확장 과제**: PART 1 비전 결과를 시리얼 대신 MQTT로 ESP32에 전달

### 자주 겪는 문제
| 증상 | 원인 | 해결 |
|---|---|---|
| 브로커 연결 실패 | 포트/주소 오류 | 브로커 주소·포트(1883) 확인 |
| 명령 안 옴 | 토픽 불일치 | 발행·구독 토픽 철자 일치 확인 |
| 끊김 반복 | keepalive 부족 | keepalive 값 늘리기 |

### 정리
이제 거리에 상관없이, 여러 대를 동시에 제어할 수 있다. 마지막으로 공유기 없이 동작하는 근거리 무선을 배운다.

---

## Lesson 12. 근거리 무선 — BLE & ESP-NOW

### 도입
WiFi·MQTT는 공유기나 인터넷이 필요하다. 현장에 네트워크가 없거나 ESP32끼리만 통신하고 싶을 때를 위한 두 가지 방식, BLE와 ESP-NOW를 배우며 통신 과정을 마무리한다.

### 핵심 개념

**BLE vs ESP-NOW**

| | BLE | ESP-NOW |
|---|---|---|
| 통신 상대 | 휴대폰·PC | 다른 ESP32 |
| 공유기 | 불필요 | 불필요 |
| 전력 | 매우 낮음 | 낮음 |
| 거리 | 10~30m | ~200m(야외) |
| 용도 | 앱으로 직접 제어 | 기기 간 직접 통신 |

**BLE 구조**
ESP32가 BLE 서버가 되어, 휴대폰 앱이 데이터를 읽고 쓴다. GATT(서비스/캐릭터리스틱)라는 구조로 데이터를 주고받는다.

**ESP-NOW 구조**
ESP32끼리 **MAC 주소**로 직접 통신한다. 공유기·인터넷이 전혀 필요 없다. 컨트롤러용 ESP32와 로봇팔용 ESP32를 무선으로 분리할 때 강력하다.

### 실습 단계
- **BLE**: ESP32 BLE 서버 설정 → 휴대폰 앱(nRF Connect)으로 좌표 전송 → 로봇팔 동작
- **ESP-NOW**: ESP32 2대 준비
  - A(컨트롤러): 가변저항 입력을 읽어 ESP-NOW로 전송
  - B(로봇팔): 수신한 값으로 서보 제어
  - Lesson의 가변저항 제어를 **무선으로 분리**하는 셈

### 최종 종합 토론
지금까지 배운 5가지 통신을 표로 정리하고, 각 시나리오에 가장 적합한 방식을 선택해본다.

| 시나리오 | 추천 통신 | 이유 |
|---|---|---|
| 실험실에서 PC로 정밀 제어 | 시리얼 | 가장 확실, 지연 적음 |
| 현장에서 폰으로 즉석 제어 | WiFi AP + WebSocket | 공유기 없이 실시간 |
| 원격지/다중 로봇 관리 | MQTT | 인터넷 통한 확장성 |
| 간단한 앱 제어, 저전력 | BLE | 공유기 불필요, 절전 |
| ESP32 간 무선 분리 | ESP-NOW | 인프라 없이 직접 통신 |

### 정리
통신은 "정답이 하나"가 아니라 "상황에 맞게 고르는 것"이다. 신뢰성·속도·거리·전력·인프라를 따져 최적 방식을 선택하는 안목이 이 과정의 최종 목표다.
